Por que Python + gRPC + Docker?

Python 3.11

Máxima legibilidade dos algoritmos. Threading nativo com Lock para concorrência. Setup simples.

03 / 12
Algoritmo 1

Relógio Lógico de Lamport

A "cola" que conecta Ricart-Agrawala ao sistema.

class LamportClock:
  def tick(self) -> int:
    # Evento local / PRÉ-ENVIO
    with self._lock:
      self._time += 1
      return self._time

  def update(self, recv_ts) -> int:
    # Ao RECEBER mensagem
    with self._lock:
      self._time = max(self._time, recv_ts) + 1
      return self._time

tick()

Chamado ANTES de enviar qualquer mensagem. Incrementa o clock local.

update(T)

Chamado ao RECEBER. Aplica max(local, T) + 1. Garante causalidade.

Thread-Safe

Lock protege o contador. Log emitido FORA do lock para não segurar I/O.

Algoritmo 2

Exclusão Mútua — Ricart-Agrawala

Regra de Prioridade

# Quem ADIA quem?
if state == HELD or (
  state == WANTED and
  (my_ts, my_id) < (their_ts, their_id)
):
  ADIAR pedido  # tenho prioridade
else:
  CONCEDER imediatamente

Ordenação Total: O par (ts, id) garante desempate determinístico — timestamps iguais são resolvidos pelo ID menor.

Estados de um Nó

 RELEASED — Não quer a CS
 WANTED — Aguardando N-1 REPLYs
 HELD — Dentro da Seção Crítica
 release_cs() → REPLY adiados
05 / 12
Comunicação

Arquitetura gRPC — Fire-and-Forget

5 RPCs unárias. Nenhum handler bloqueia. REPLY é proativo.

RequestAccess Pedir permissão para CS
ReplyAccess Conceder permissão (proativo)
Election Bully: contatar IDs maiores
Coordinator Anunciar novo líder
Heartbeat Ping/Pong vivacidade

O Problema

Se RequestAccess bloqueasse esperando a CS, o ThreadPool do gRPC esgotaria → deadlock.

A Solução

REQUEST e REPLY são RPCs separadas. O handler registra e retorna Ack imediato. Ao sair da CS, dispara REPLY proativamente em thread.

06 / 12
Concorrência

Sincronização e Thread-Safety

O gRPC usa ThreadPool (20 workers). Múltiplas threads alteram estado simultaneamente.

LamportClock

threading.Lock em tick() e update().

Log emitido fora do lock (evita I/O dentro da região crítica).

RicartAgrawala

Um único lock protege: state, request_ts, _pending_replies, _deferred.

Transição WANTED→HELD é atômica dentro de on_reply().

BullyElection

Lock cobre leader_id e election_in_progress.

on_election() seta flag atomicamente para impedir threads duplicadas.

Zero deadlocks
Zero race conditions
20 workers no ThreadPool

Algoritmo 3

Eleição de Líder — Bully

Fluxo do Algoritmo

  • Heartbeat falha → Líder caiu
  • Envia ELECTION a todos de ID maior
  • Se alguém responde Ack → "OK, eu assumo"
  • Se ninguém responde → sou o líder
  • Anuncia COORDINATOR a todos

Insight: O sucesso da RPC (Ack) já serve como "OK" do Bully. Não precisamos de mensagem extra.

08 / 12
Tolerância a Falhas

Detecção e Recuperação Automática

Mecanismo de Detecção

Heartbeat Loop

Thread dedicada pinga o líder a cada 2s. Timeout de 2s na RPC. Se falha → report_leader_failure().

Recuperação

Após detectar falha, dispara run_election(). O maior ID vivo assume. COORDINATOR é enviado com retry (3x).

Demonstração ao Vivo

# Terminal B — simula falha
$ docker stop node5

# Nos logs (2-3 segundos depois):
👑 Node 3 | Falha do lider detectada!
👑 Node 4 | SOU O NOVO LIDER!

Tempo de recuperação: ~4 segundos (2s timeout + 2s eleição)
